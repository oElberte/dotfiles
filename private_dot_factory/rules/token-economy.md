# Token Economy Rules

- Be concise, direct, and high-signal.
- Remove filler, repeated context, unnecessary praise, and obvious explanations.
- Prefer compact bullets, checklists, and command-first answers.
- Do not compress code, commands, paths, errors, stack traces, model names, env vars, config keys, or architecture decisions where nuance matters.
- Keep final coding responses to: Changed, Validated, Risks/uncertainties, Next action only if needed.
- Use `frun` for shell commands when available; fallback to raw commands if unavailable.

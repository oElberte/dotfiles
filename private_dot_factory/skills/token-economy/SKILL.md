# Token Economy

Be concise without losing precision.

## Rules

- No filler, repeated context, or unnecessary praise.
- Prefer compact bullets and direct commands.
- Keep explanations proportional to risk and complexity.
- Preserve exact code, commands, paths, errors, stack traces, package names, model names, env vars, and config keys.
- Use professional tone; do not use gimmick speech.
- For reviews, use one actionable line per finding when safe.
- For commit messages, use Conventional Commits; body only when the why is not obvious.

## Final Coding Response

1. Changed
2. Validated
3. Risks/uncertainties
4. Next action, only if needed
